动机

折腾了几天 Ansible, 理由是 SSH 到远程机器的连接经常断, 而且我还要部署两台机器, 所以不如让 Ansible 一口气把活干完。

我通常写个脚本直接 make + scp + ssh 完成任务，但这种脚本不好管理。为什么不一开始就用 Ansible 呢？因为不好入门，我个人感觉官方文档非常难懂。但写过几个配置之后，我发现 Ansible 就是自动化工具的终极形态，我们都应该学习 Ansible。

参考

先读一遍 Ultimate Ansible Bootcamp，官方文档不适合入门。

-v

-v, -vv, -vvv, -vvvv 分别表示不同级别的 debug 信息

配置 hosts

编辑 /etc/ansible/hosts

[itx]
192.168.42.2

[r2s]
192.168.42.3

[tv]
192.168.42.233

[组名] 标记了组名，all 组包括所有 ip

测试：

ansible all -m ping
ansible r2s -m ping

ad hoc 命令

ad hoc 是 for this purpose 的意思，ad: to, hoc: this

ansible itx -a "echo hi"
192.168.42.2 | CHANGED | rc=0 >>
hi

返回值

这里有一个返回值的检验过程：

---
- hosts: localhost
  tasks:
  - debug:
      msg: Testing a binary module written in Rust
  
  - debug:
      var: ansible_system
  
  - name: ping
    ping:
  
  - name: Check out architecture
    ansible.builtin.shell: /usr/bin/uname -p
    register: arch

  - debug: var=arch.stdout_lines

  - assert:
      that:
      - > 
          arch.stdout_lines[0] == "x86_64"

我想执行目标机器上某个工具的返回值，那么就需要用 - debug 模块把返回值打印出来，比如，我之所以用 arch.stdout_lines[0] 是因为我看到 stdout_lines 是一个数组:

TASK [debug] **********************************************************************************************************************************************************************
ok: [localhost] => {
    "arch.stdout_lines": [
        "x86_64"
    ]
}

如果你要检查自己写的工具的输出，那么要确保你的工具输出是一个 json, 这里是一个 Rust 的返回值的例子

    exit_json(Response {
        msg: format!("Hello, {}!", module_args.name.as_str()),
        changed: true,
        failed: false,
    });

role

这里有一堆例子，你可以按照上面的例子布置 ansible 项目的结构，也可以用命令行生成：

mkdir roles
ansible-galaxy init --offline --init-path=roles  apache

参考优先参考 Ultimate Ansible Bootcamp，里面的例子可以直接照抄。

ansible 的 role 跟插件一样，可以直接安装：

 ansible-galaxy install $author.$role

举个我写的例子：

假设你的主机在 r2 组，那么你首先在项目根目录创建一个 group_vars 的目录，里面创建一个 r2.yml 文件，名字比如和组名一样，然后填入配置

然后，创建一个叫做 webserver.yml 的文件，其中引用了 nginx 这个 role

最后，创建一个 role, 比如说叫做 nginx，然后写入命令，模板，等等，这些很简单。

nginx 目录之下有几个常用目录：

  • templates： 模板文件，jinja2 格式，搞过前端应该比较熟悉
  • files: 原封不动的文件就放在这
  • tasks: 命令
  • handlers: 相当于 callback, 在 tasks 的命令上通过 notify 字段调用此目录的 callback

我写的其中一个 tasks/main.yml，安装一个叫做 something 的服务：

---
- name: Install something
  copy:
    src: something
    dest: /usr/local/bin/
    mode: 0755

- name: Install systemd unit file
  template:
    src: something.service.j2
    dest: /lib/systemd/system/something.service

- name: Ensure /etc/something exists
  file: path=/etc/something state=directory

- name: Copy something configuration file
  template:
    src: server.json.j2
    dest: /etc/something/config.json

- name: Start something service
  systemd:
    name: something
    state: started
