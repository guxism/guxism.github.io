Nestjs 简介

用了一段时间 Nestjs， 说说心得。在简单的需求方面 Nestjs 和其他框架别无二致，本文先简单介绍 Nestjs 的用法，然后从一个业务需求入手，讲到 Nestjs 的高级特性。

简单用法

项目的最初阶段我是直接套用它的模板：

nest new project-name

然后在 src 目录加一个子目录，用来放业务逻辑，当然这只是基本的代码管理，代码放哪都可以，VSCode 已经很智能，知道你要引用哪些定义，直接就帮你把 import 补上。

进阶

我认为稍微进阶的标志是使用他们的命令行工具，这一点有些好处，就是知道你使用这个框架的什么组件。当然前提是你认可这一框架，只有认可了你才会把时间投入到这上面。对我来说这些命令不是某个框架特有的，很多框架都有相似的命令，我们了解一个之后，可以举一反三。

生成一个 Module:

nest g module cats

生成一个目录 cats, 里面只有一个文件：

import { Module } from '@nestjs/common';

@Module({})
export class CatsModule {}

生成这个 Module 并不说明什么，它处于闲置状态，使用它的方式是 import 它，但要 import，首先就要标注它的 exports

但由于这个 Module 什么都没有，所以没什么好 export 的。因此此时我们要创建 controller 和 provider.

controller 的意义是提供针对某一个具体路径的请求的处理方法，而 provider 的意义是提供这些处理方法

nest g controller cats
nest g service cats

这两行指令把 cat.module.ts 改成了：

import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class CatsModule {}

注意我并没有手动添加代码。接着就要让其他模块引用 CatsModule

@Module({
  imports: [CatsModule],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}

但其实它已经被 import 了

这时我们要到 CatsController 去验证 CatsModule 的可用性

@Controller('cats')
export class CatsController {
  @Get()
  saySomething() {
    return 'hail nestjs!\n';
  }
}

$ curl localhost:3002/cats
hail nestjs!

把 saySomething 挪到 provider 去：

// cats.service.ts
@Injectable()
export class CatsService {
  hail() {
    return 'hail nestjs!\n';
  }
}
// cats.module.ts
@Controller('cats')
export class CatsController {
  constructor(private catsService: CatsService) {}
  @Get()
  saySomething() {
    return this.catsService.hail();
  }
}

这也是 proxy pattern 的体现

关于 CatsService 有几个关键的概念：

  1. @Injectable() 标注了 CatsService 是一个 provider
  2. 依赖注入，注意这一个概念其实并不是委托那么简单，它指的是仅在使用的时候才初始化真实的 object, 这一点反而比委托更为重要，这也就是为什么 Nestjs 官网说 dependency injection is an inverse of control(IOC) technique, wherein you delegate instantiation of dependencies to the IOC container

Standard providers: 实际上 providers: [CatsService] 是以下语法的简化版(short hand)

providers: [
	{
		provide: CatsService
		useClass: CatService
	}
]

接着到 Custom Provider, 后者就是为了使用以上复杂的语法

providers: [
	{
		provide: CatsService
		useClass: mockCatsService
	}
]

不难理解，CatsService 映射 mockCatsService, 注意到 useClass 这种标识符，不难想象，mockCatsService 必然是 class

存在这种情况: 使用的不是 class(例如 mockCatsService ), 而是实例/值，这样的话，就需要用到 useValue, 看了一波文档，发现没说清楚，但是看例子一目了然

首先在 app.module.ts 指定 providers:

@Module({
  imports: [CatsModule],
  controllers: [AppController],
  providers: [
    AppService,
    {
      provide: 'PORT',
      useValue: 3003,
    },
  ],
})
export class AppModule {}

某种程度上，你要认为 providers 最终会变成一张表，还要注意到这里用到了修饰器，跟宏一样，如果你能看到 AppService 字样，那么就会有一个 'AppService', 理由是如果按照 class 的思路来的话，'PORT' 实际应该对应一个 PORT。当然我这么论证是不对的，因为我没仔细去研究 @Module() 的机制，我只是找一个容易接受的切入点。

接着看如何使用：

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  const port = app.get('PORT');
  await app.listen(port);
}
bootstrap();

app.get() 意味着 app 要么是一个字典，要么就是字典的代理，总而言之必然有一张表，不是表，是列表也无所谓，但是哈希表查询的效率更高。获取的值是 3003。

接着看一些更高级的用法，由于我看文档看得不是很明白，所以看了这篇 https://dev.to/nestjs/advanced-nestjs-dynamic-providers-1ee

最好的学习方式是照做：

 src   master  nest g module Logger
CREATE logger/logger.module.ts (83 bytes)
UPDATE app.module.ts (444 bytes)
 src   master ±  nest g service Logger
CREATE logger/logger.service.spec.ts (460 bytes)
CREATE logger/logger.service.ts (90 bytes)
UPDATE logger/logger.module.ts (163 bytes)

写入：

// src/logger/logger.service.ts

@Injectable({
  scope: Scope.TRANSIENT,
})
export class LoggerService {
  private prefix?: string;

  log(message: string) {
    let formattedMessage = message;

    if (this.prefix) {
      formattedMessage = `[${this.prefix}] ${message}`;
    }

    console.log(formattedMessage);
  }

  setPrefix(prefix: string) {
    this.prefix = prefix;
  }
}

Scope.TRANSIENT 意味着每一个注入到其他类去的 LoggerService 实例都是独一无二的，而不是 singleton。默认是 singleton

要使用这一个 Service(也就是 provider)，就要在相关的 logger.module.ts 中把它导出：

@Module({
  providers: [LoggerService],
  exports: [LoggerService],
})
export class LoggerModule {}

然后将 LoggerModule 用在 app.service.ts 上

import { Injectable } from '@nestjs/common';
import { LoggerService } from './logger/logger.service';

@Injectable()
export class AppService {
  constructor(private readonly logger: LoggerService) {
    this.logger.setPrefix('AppService');
  }
  getHello(): string {
    this.logger.log('Hello World');
    return 'Hello World!';
  }
}

Service 这一个词有些误导人，但可以把它理解为可复用代码

再执行 curl:

curl localhost:3003

下一步就是将 this.logger.setPrefix 用修饰器替代掉

创建 src/logger/logger.decorator.ts

export const prefixesForLoggers: string[] = new Array<string>();

export function Logger(prefix = '') {
  if (!prefixesForLoggers.includes(prefix)) {
    prefixesForLoggers.push(prefix);
  }
  return Inject(`LoggerService${prefix}`);
}

这是一个装饰器，被 evaluate 之后返回一个 Inject()

创建 src/logger/logger.providers.ts

function loggerFactory(logger: LoggerService, prefix: string) {
  if (prefix) {
    logger.setPrefix(prefix);
  }
  return logger;
}

function createLoggerProvider(prefix: string): Provider<LoggerService> {
  return {
    provide: `LoggerService${prefix}`,
    useFactory: (logger) => loggerFactory(logger, prefix),
    inject: [LoggerService],
  };
}

export function createLoggerProviders(): Array<Provider<LoggerService>> {
  return prefixesForLoggers.map((prefix) => createLoggerProvider(prefix));
}

这里有些不好理解的地方，可以看出，关键函数是 createLoggerProviders, 这个函数返回一组 provider, 每个 provider 的结构如下：

{
    provide: `LoggerService${prefix}`,
    useFactory: (logger) => loggerFactory(logger, prefix),
    inject: [LoggerService],
};

我们知道 provide 就是一个索引，但 useFactory 和 inject 是很陌生的概念。

接下来看如何使用关键函数 createLoggerProviders:

const loggerProviders = createLoggerProviders();

@Module({
  providers: [LoggerService, ...loggerProviders],
  exports: [LoggerService, ...loggerProviders],
})
export class LoggerModule {}

首先创建一组 providers, 然后放在 @Module() 修饰器的参数列表中

把 AppService 改成：

@Injectable()
export class AppService {
  constructor(@Logger('AppService') private logger: LoggerService) {}
  getHello(): string {
    this.logger.log('Hello World');
    return 'Hello World!';
  }
}

这么做省了哪些步骤？

-  constructor(private readonly logger: LoggerService) {
-    this.logger.setPrefix('AppService');
-  }
+  constructor(@Logger('AppService') private logger: LoggerService)

确实，代码得到了更好的封装

但其实还能做得更好，将 src/logger/logger.providers.ts 改成：

-@Module({
-  providers: [LoggerService, ...loggerProviders],
-  exports: [LoggerService, ...loggerProviders],
-})
-export class LoggerModule {}
+export class LoggerModule {
+  static forRoot(): DynamicModule {
+    const prefixedLoggerProviders = createLoggerProviders();
+    return {
+      module: LoggerModule,
+      providers: [LoggerService, ...prefixedLoggerProviders],
+      exports: [LoggerService, ...prefixedLoggerProviders],
+    };
+  }
+}

也就是将 @Module() 去掉，换成一个 forRoot() 函数，返回一个 DynamicModule, 如此对比 @Module() 就是 static module

注意 forRoot() 只是一个惯用名，可以用其他名字：

can have any arbitrary name, but by convention we should call it either forRoot() or register()

更新 app.module.ts:

 @Module({
-  imports: [CatsModule, LoggerModule],
+  imports: [CatsModule, LoggerModule.forRoot()],
   controllers: [AppController],
   providers: [

以上我们能获取哪些信息？

  1. import 是用来 import module 的，imports 这一键(field) 既可以写死的，也可以是动态生成的，而这种动态生成看似和写死没什么区别，实际上一旦允许列表元素在编译的时候执行，就意味着可以和外界交互，这实际上是打开了一扇门，多了许多可能性。

另外一个例子：ConfigModule

看官方文档的例子：https://github.com/nestjs/nest/blob/master/sample/25-dynamic-modules/src/main.ts

src/app.module.ts

@Module({
  imports: [ConfigModule.register({ folder: './config' })],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}

此处引用了 ConfigModule.register() 的产物，意味着在整个 AppModule(包括了 app.controller.ts, app.service.ts) 都可以使用 ConfigModule

比如 app.service.ts:

@Injectable()
export class AppService {
  private helloMessage: string;

  constructor(configService: ConfigService) {
    this.helloMessage = configService.get('HELLO_MESSAGE');
  }

  getHello(): string {
    return this.helloMessage;
  }
}

这时候我们要追踪到 ConfigModule, 看它如何实现 register()

首先看 src/config/config.module.ts

export interface ConfigModuleOptions {
  folder: string;
}

@Module({})
export class ConfigModule {
  static register(options: ConfigModuleOptions): DynamicModule {
    return {
      module: ConfigModule,
      providers: [
        {
          provide: CONFIG_OPTIONS,
          useValue: options,
        },
        ConfigService,
      ],
      exports: [ConfigService],
    };
  }
}

看返回值：

{
	module: ConfigModule,
    providers: [
	    {
          provide: CONFIG_OPTIONS,
          useValue: options,
        },
        ConfigService,
      ],
      exports: [ConfigService],
};

可以看到有两个 provider:

  • CONFIG_OPTIONS
  • ConfigService

我们已经看到第二个的使用场景：

@Injectable()
export class AppService {
  private helloMessage: string;

  constructor(configService: ConfigService) {
    this.helloMessage = configService.get('HELLO_MESSAGE');
  }

  getHello(): string {
    return this.helloMessage;
  }
}

第一个比较不好理解：

constructor(@Inject(CONFIG_OPTIONS) options: ConfigOptions) {
    const filePath = `${process.env.NODE_ENV || 'development'}.env`;
    const envFile = path.resolve(__dirname, '../../', options.folder, filePath);
    this.envConfig = dotenv.parse(fs.readFileSync(envFile));
  }

怎么说呢，@Inject(CONFIG_OPTIONS) 指定了 options 就是以下配置的 options:

{
    provide: CONFIG_OPTIONS,
    useValue: options,
},

这个 options 是 ConfigModule.register({ folder: './config' }) 中的

{
	folder: './config'
}

我们也可以看到这一行：

const envFile = path.resolve(__dirname, '../../', options.folder, filePath);
