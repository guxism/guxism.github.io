Relearn CMake

CMake is hard, I have written a lot of cmake but still find it scary. But maybe every build system is hard, because it's a matter of organization and management. And though one might have written a lot of building manifests, he is not writing them every day. So the best practice is to sort it out and when the time comes, go check the notes and get ready to spend hours in it. This article is about how to write a CMakeLists.txt from scratch.

  2. Project Name

Cmake Version Requirement

First off, this is a must, some advanced new cmake features require higher version of cmake, it is 2022, so let's use a higher version.

cmake_minimum_required(VERSION 3.8)

Project Name

Project name is whatsoever you like.

project(morg)

Language Standard

I am using C++20 so I must specify the standard, it does the same thing like g++ -std=c++2a.

set(CMAKE_CXX_STANDARD 20)

Include GoogleTest

To include GoogleTest, there are a few methods but I prefer this one below, because it's simpler:

include(FetchContent)
FetchContent_Declare(
  googletest
  URL https://github.com/google/googletest/archive/e2239ee6043f73722e7aa812a459f54a28552929.zip
)
# For Windows: Prevent overriding the parent project's compiler/linker settings
set(gtest_force_shared_crt ON CACHE BOOL "" FORCE)
FetchContent_MakeAvailable(googletest)

I took them straightly from GoogleTest official document Quickstart: Building with CMake

Include Other Libraries

To include other libraries is similar to including GoogleTest, for example, I want to include moodycamel::ConcurrentQueue, I go to the repository, and consider 22c78daf65d2c8cce9399a29171676054aa98807 to be a reasonably stable commit, so I append the commit hash to the URL as below:

FetchContent_Declare(
  concurrentqueue
  URL https://github.com/cameron314/concurrentqueue/archive/22c78daf65d2c8cce9399a29171676054aa98807.zip
)
FetchContent_MakeAvailable(concurrentqueue)

Then include its header, go check its CMakeLists, to see how it exposes its interfaces:

target_include_directories(${PROJECT_NAME}
        INTERFACE
                $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}>
                $<INSTALL_INTERFACE:${CMAKE_INSTALL_INCLUDEDIR}/${PROJECT_NAME}/>
)

It means that if I want to use the source of the library to build my own project, I should find its header in the root directory of the source, in this case, I should write #include <concurrentqueue.h>. But if the source has been compiled into a library, I should write #include <concurrentqueue/concurrentqueue.h> instead.

Build Our Own Source

Building our own source is simpler:

set(MORG_INCLUDE_DIR ${PROJECT_SOURCE_DIR}/include)
set(MORG_LIB morg_lib)
add_library(${MORG_LIB} INTERFACE)
target_include_directories(${MORG_LIB} INTERFACE
  ${MORG_INCLUDE_DIR}
)

Link GoogleTest to the tests:


enable_testing()
file(GLOB_RECURSE MORG_TESTS_FILES ${CMAKE_CURRENT_SOURCE_DIR}/tests/*.cpp)
set(MORG_TEST morg_test)
add_executable(${MORG_TEST}
    ${MORG_TESTS_FILES}
    #${CMAKE_CURRENT_SOURCE_DIR}/tests/main.cpp
    )
target_link_libraries(${MORG_TEST} ${MORG_LIB} gtest_main concurrentqueue)

include(GoogleTest)
gtest_discover_tests(${MORG_TEST})

Build executables:

set(MORG_EXEC morg)
add_executable(${MORG_EXEC}
    ${CMAKE_CURRENT_SOURCE_DIR}/morg.cpp)
# https://stackoverflow.com/a/29871891/9639179
set(THREADS_PREFER_PTHREAD_FLAG ON)
find_package(Threads REQUIRED)
target_link_libraries(${MORG_EXEC} ${MORG_LIB} concurrentqueue Threads::Threads)

Conclusion

There is nothing more to say, I spent 2 hours in writing these 50 lines of code, a lot of time is spent in downloading GoogleTest and its usage, it turns out that FetchContent is a new mechanism of package management. It will be a lot easy if I use Rust. I have this feeling that C++ will fail eventually. Knowing how to write a CMakeLists.txt is not a honor, but a modern primitive. But if I have to do that again, this article will be my guidance.
