How bridges work

This world is built with crap among usable tools. We must be strong to hold ourselves -- that is, we must literally read the code of a bridge project to understand how the kind of thing work. Let's read this: https://github.com/ChainSafe/chainbridge-solidity

Some prerequisite concepts

Lots of .sol files, but we will start with contracts/interfaces/IBridge.sol

The interface is simple:

interface IBridge {
	function _chainID() external returns (uint8);
}

the next sol: contracts/interfaces/IERCHandler.sol

interface IERCHandler {

	function setResource(bytes32 resourceID, address contractAddress) external;

	function setBurnable(address contractAddress) external;

	function setDecimals(address contractAddress, uint8 srcDecimals, uint8 destDecimals) external;

	function withdraw(address tokenAddress, address recipient, uint256 amountOrTokenID) external;

}

Something must be said before we delve in, a bridge acts like a proxy, it is not the target, it is a forwarder, a forward process the input from users, and send the input to another processor. That processor is attached to the bridge in some way. Usually the bridge administrator just registers the address of the said sibling processor inside the bridge. Then we call the address a handler. Remember how we handle signals in C language?

signal(SIGINT,sig_handler);

The code above registers the sig_handler to take care of the SIGINT signal. When a SIGINT reaches the process, sig_handler is called. Here, sig_handler is a function, in some sense, it doesn't matter what handlers are, as long as you can use it to track down a processing entity, a function, a HTTP-server, a contract.

Just remember handle is a jargon in the bridge domain, and it has a special meaning in Web3 development, it means something that will take over the data, that is, to handle the data. Functions can handle data, locally, but they already have a name: Function. Although no one says the thing in public, but we know in our hearts that "handlers" are external data processors.

Smart contracts can be handlers.

interface IGenericHandler {

	function setResource(bytes32 resourceID, address contractAddress, bytes4 depositFunctionSig, bytes4 executeFunctionSig) external;

}
  • one example of resourceID is a hex string which is not an address in the trait, but something that is more like UUID to identify a kind of resource from the others.
  • the second arguments is a contract address. When a deposit is made, a function will be called in the contract.
  • for now we just skip the last two arguments. But it's not hard to tell that depositFunctionSig is something that will be called in the deposit function of the bridge.

Bridge's deposit

    function deposit(uint8 destinationChainID, bytes32 resourceID, bytes calldata data) external payable whenNotPaused {
        require(msg.value == _fee, "Incorrect fee supplied");

        address handler = _resourceIDToHandlerAddress[resourceID];
        require(handler != address(0), "resourceID not mapped to handler");

        uint64 depositNonce = ++_depositCounts[destinationChainID];

        IDepositExecute depositHandler = IDepositExecute(handler);
        depositHandler.deposit(resourceID, destinationChainID, depositNonce, msg.sender, data);

        emit Deposit(destinationChainID, resourceID, depositNonce);
    }

Let's focus on this: chainbridge-solidity/contracts/handlers/ERC20Handler.sol

    function deposit(
        bytes32 resourceID,
        uint8   destinationChainID,
        uint64  depositNonce,
        address depositer,
        bytes   calldata data
    ) external override onlyBridge {
        bytes   memory recipientAddress;
        uint256        amount;
        uint256        lenRecipientAddress;

        (amount, lenRecipientAddress) = abi.decode(data, (uint, uint));
        recipientAddress = bytes(data[64:64 + lenRecipientAddress]);

        address tokenAddress = _resourceIDToTokenContractAddress[resourceID];
        require(_contractWhitelist[tokenAddress], "provided tokenAddress is not whitelisted");

        if (_burnList[tokenAddress]) {
            burnERC20(tokenAddress, depositer, amount);
        } else {
            lockERC20(tokenAddress, depositer, address(this), amount);
        }

        _depositRecords[destinationChainID][depositNonce] = DepositRecord(
            tokenAddress,
            destinationChainID,
            resourceID,
            recipientAddress,
            depositer,
            amount
        );
    }

What do we have here? The irritating thing is that we can't debug it in a convenient way

  1. the step that deduce amount and tokenAddress from the data and abi
  2. get the token address
  3. check if the token address in in the white list
  4. check if the token is burnable, if yes, burn it, else just lock it
  5. put what just happened into a record

if it burns, how?

read contracts/ERC20Safe.sol (which is inside the handler contract)

function burnERC20(address tokenAddress, address owner, uint256 amount) internal {
	ERC20Burnable erc20 = ERC20Burnable(tokenAddress);
	erc20.burnFrom(owner, amount);
}

burn an amount of money from owner

then we must go to "@openzeppelin/contracts/token/ERC20/ERC20Burnable.sol" (which is inside the token contract)

function burnFrom(address account, uint256 amount) public virtual {
        uint256 decreasedAllowance = allowance(account, _msgSender()).sub(amount, "ERC20: burn amount exceeds allowance");

        _approve(account, _msgSender(), decreasedAllowance);
        _burn(account, amount);
    }

oh, so, first _approve, the caller approves the contract to use a certain sum of money from him.

then _burn, we are very interesting in it:

function _burn(address account, uint256 amount) internal virtual {
        require(account != address(0), "ERC20: burn from the zero address");

        _beforeTokenTransfer(account, address(0), amount);

        _balances[account] = _balances[account].sub(amount, "ERC20: burn amount exceeds balance");
        _totalSupply = _totalSupply.sub(amount);
        emit Transfer(account, address(0), amount);
    }

what it does:

  1. _beforeTOkenTransfer is a hook that is called before any tranfer of tokens
  2. the balance of the account subtracts the amount that is to be burned
  3. the totalSUpple decrease amount
  4. the Event Transfer is emitted

this is how approve works:

    function _approve(address owner, address spender, uint256 amount) internal virtual {
        require(owner != address(0), "ERC20: approve from the zero address");
        require(spender != address(0), "ERC20: approve to the zero address");

        _allowances[owner][spender] = amount;
        emit Approval(owner, spender, amount);
    }

If it can't burn, it locks

function lockERC20(address tokenAddress, address owner, address recipient, uint256 amount) internal {
        IERC20 erc20 = IERC20(tokenAddress);
        _safeTransferFrom(erc20, owner, recipient, amount);
    }

We can see that another contract is initiated from tokenAddress, it transfer an amount of tokens from the owner to the recipient
