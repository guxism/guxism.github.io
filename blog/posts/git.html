进阶操作

git rebase

main 和 bugFix 两个分支, 其中 bugFix 修好了 bug, 想将 bugFix 的提交放在 main 的前面

git checkout -b bugFix
# fix bugs
git add .
git commit -m "fixed"
git checkout main
# while bugs are being fixed by bugFix, someone pushes new commits to main
git add .
git commit -m "something stirs in main",
# keep bugFix ahead of main
git checkout bugFix
git rebase main

适用于多人协作项目, 保持自己的修改是最新提交

Detach HEAD

HEAD 是一个指针, 全局只有一个 HEAD, HEAD 的作用是提供基准点, 以便进行 git reset HEAD^ 这样的操作.

一般来说，HEAD 指向 branch，detach HEAD 意味着 HEAD 不再指向 branch 而是指向某个特定的 commit

git checkout COMMIT-1

Reference

  • COMMIT^ COMMIT 之前的提交
  • HEAD^ 往上一个提交
  • HEAD~N 往回第 N 个提交
  • HEAD~1 == HEAD^
  • HEAD~2 == HEAD^^
  • git branch -f main HEAD: 将 main 标签移动到 HEAD 的位置

git revert

创建一个仓库, 创建一个 a.txt, 提交, git revert HEAD 创建一个新提交, 这个提交导致 a.txt 消失

test ➤ git init .                                                              
Initialized empty Git repository in /tmp/test/.git/
test ➤ echo 1 > a.txt 
test ➤ git add .     *
test ➤ git commit -m "initial"  
[master (root-commit) 0700fdb] initial
 1 file changed, 1 insertion(+)
 create mode 100644 a.txt
test ➤ git revert HEAD
Removing a.txt
[master c1c1280] Revert "initial"
 1 file changed, 1 deletion(-)
 delete mode 100644 a.txt
test ➤ git log        
test ➤ ls             
test ➤

git cherry-pick

git cherry-pick 类似与 git rebase, 不同的是 git cherry-pick 不需要指定分支, 而是要指定提交. 例如, 将 a 分支的 C1, b 分支的 C2, c 分支的 C3 添加到当前分支上, 则执行:

git cherry-pick C1 C2 C3

git rebase -i

有选择余地的 git rebase

git submodule

git submodule update --init

出现某个子模块(比如 contrib/libcxx) 更新失败之后

git submodule update --remote contrib/libcxx

或者

find contrib -maxdepth 1 -type d | xargs git submodule update --remote

可能出现不可解决的问题:

git submodule foreach git reset --hard

有时候不同的分支可能有不同的子模块, 切换之后出现 untracked files, 直接 rm 就行了, 切换之后记得执行:

git submodule update
