For those who are new to FP I recommand this book Professor Frisby's Mostly Adequate Guide to Functional Programming. Haskell is not for everybody, as it involves some serious mathematics which adds to the difficulty of learning, we had better pick something easier to start with. During my reading, I find that to survive the book, one must have a preliminary knowledge of Javascript's queer and obscure features that we don't frequently encounter in our daily life.

Arrow functions

const always = (a, b) => a

always is an arrow function.

Arity

> const fn = (a, b) => a
undefined
> fn.length
2

bind()

Javascript assumes functions should have a context, whereas this does not apply to arrow functions, for functions that are declared with the function keyword, it is possible to set their context to null, it's like cutting them off from the environment.

> function needy(y) { return this.x + y; }
undefined
> needy(2)
NaN
> const context = { x: 40 };
undefined
> const fulfilled = needy.bind(context);
undefined
> fulfilled(2)
42
> const fulfilledFixture = needy.bind(context, 2);
undefined
> fulfilledFixture()
42
> function unsatisfied(y, z, h, i, j) { return this.x + y + z + h + i + j; }
undefined
> const massivelyFulfilled = unsatisfied.bind(context, 1, 2, 3, 4, -8)
undefined
> massivelyFulfilled()
42

Cutting the functions off from their context makes thing easy, no need to worry about Javascript's over-complicated and inconherent semantics of this. I myself will never allow my code to rely on this.

New

New is the operator to instantiate classes; A function can act as a constructor to a class that has not been defined, as long as the function is configuring the this object.

> function Foo(bar1, bar2) {
...     this.a = bar1;
...     this.b = bar2;
... }
undefined
> obj = new Foo(1, 2);
Foo { a: 1, b: 2 }
> obj.a + obj.b
3

The this object is constructed by the new operator from the Foo and then assigned to obj.

Call

The call() method calls a function with a given this value and arguments provided individually

Look at this example from MDN docs:

> function Product(name, price) {
...   this.name = name;
...   this.price = price;
... }
undefined
> 
> function Food(name, price) {
...   Product.call(this, name, price);
...   this.category = 'food';
... }
undefined
> const cheese = new Food('feta', 5);
undefined
> cheese
Food { name: 'feta', price: 5, category: 'food' }

It's hard to believe that now cheese has three properties, but think this way: Product.call actually sets this's name property to be price, and its side affect is to mutate the this object, which then leaves the function Product.call, and returns back to constructor Food's environment, and gains a new property called category.

To use call(), the caller should pass the this object to the call() function, and think that inside the call() function, the this object is mutated, after returning from the call(), the caller will have a new this object. You might as well think of this as a reference parameter. It's like in sophisticated C programming, peoole always define their functions with a context parameter as the first argument, so as not to rely on global variables, so as to be reentrant. But it's not like the said C style, in that this is never declared as the first parameter in any function. I will say no more, the incoherence of Javascript is a mind killer.

Anyway, if you set the first argument to null, you don't need to worry about this anymore.

Implementing Currying

function curry(fn) {
    const arity = fn.length;
    
    return function $curry(...args) {
        if (args.length < arity) {
            return $curry.bind(null, ...args);
        }
        return fn.call(null, ...args);
    }
}

Let's pass arrow function (a, b) => a to curry() and walk through it:

  • arity is 2
  • curry((a, b) => a) return a function with the following definition:
function mystical_func(...args) {
    if (args.length < 2) {
        return some_func.bind(null, ...args);
    }
    return fn.call(null, ...args);
}
  • Say we invoke mystical_func() with ...args being the number 2, then args.length is 1, then mystical_fun() returns another function. Remember what we learned about bind(), with the first argument set to null, mystical_func() is total free from this, which is great, but does not help if we have never use this in the definition. What matters is that the generated function, which is called bound function, takes args as its preset arguements, or in another word, pre-specified initial arguements. To call this new function we must be aware that we are passing arguments to feed the origin function, in this case, (a, b) => a, but the arguments are not mapped to its parameters by starting from the first one, but from the leftest one that has not been initialized in the bound function.
    • More specifically, mystical_func(2) returns a bound mystical_func() that has a preset first argument as 2. Now name this bound function by bound_mystical_func(), to call it with the argument 3, is equivalent to call mystical_func with two arguments 2, 3 in a line.
  • We don't care about fn.call because it's a fallback for those who misuse the curry() function.

Verdict

The book Mostly Adequate Guide to Functional Programming places the implementation of curry in Appendix, which seems to me not so adequate, simply knowing the concept of currying is not enough, we won't made it to the next chapter without knowing its implementation. The process of reducing the number of arguments is called partial application, somewhat similar to build pattern which configures an object step by step, in the case of FP it's argument by argument. The pursuit of functional purity demands one input to one output, to achieve that goal we must twist Javascript variadic functions into a chain of bound functions as explained above.
