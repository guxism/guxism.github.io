-fno-common

编者按：这些问题其实只要有浏览器/书在手都能快速解决，但有时候需要把它们装在脑袋里发酵，否则浪费了过往的经验。这是几个月前和朋友讨论的时候写的东西，问题是”全部使用默认参数编译构建，无告警无错误，却链接了错误的符号“。这个问题涉及到符号解析(symbol resolution)，在这个过程中，对于多个同名变量，解析器首先选择 strong symbol，未初始化变量属于 weak symbol, 但朋友发现这种情况下编译器报错为 multiple definition of x, x 有两个符号，一个已经初始化，一个没有。

太长不看：现代编译器对未初始化的变量进行默认初始化，放在 .bss section，而老版本编译器将其归类为 Common 变量，在符号解析的时候不会优先选择。

我注意到 arch linux 的 x 是 B 类型，意味着 x 在 .bss section，也就意味着它已经初始化，因为 CSAPP Practice Problem 7.2 前面有一段话讲到：

on the other hand, if x is initialized to zero, then it is a strong symbol, so the compiler can confidently assign it to .bss

我在 Ubuntu 20.04 上编译的 bar3.o 符号表为：

$ nm bar3.o
0000000000000000 T f
0000000000000004 C x

C 意味着 COMMON

Common symbols are uninitialized data -- https://linux.die.net/man/1/nm

然后我发现 -fno-common 选项：

The -fno-common option specifies that the compiler should instead place uninitialized global variables in the BSS section of the object file. ---https://man7.org/linux/man-pages/man1/gcc.1.html

添加这个选项后：

bar3.o: bar3.c
	gcc -o $@ -c $^ -fno-common

结果为：

nm bar3.o
0000000000000000 T f
0000000000000000 B x

B 意味着 x 在 .bss section

最后找到一个类似的问题： https://github.com/FreeSpacenav/spacenavd/issues/17 ：[master] Fails to compile with compile flag -fno-common (that is default for GCC >=10)

注意到你的 arch linux 编译器是 11 版本，而我的是 9，你的 gcc 默认开启了 -fno-common

那么也就意味着 foo3.c 和 bar3.c 的 x 都是 strong symbol, 那么比如违反 680 页的 rule1: mutlple strong symbols with the same name are not allowed

在 Ubuntu 20.04, gcc 9 的条件下，加上 -fno-common 可以复现 arch linux ， gcc 11 的错误

all: a.out foo3.o bar3.o

a.out: bar3.o foo3.o
	gcc -o $@ $^
foo3.o: foo3.c
	gcc -o $@ -c $^ -fno-common
bar3.o: bar3.c
	gcc -o $@ -c $^ -fno-common

编译结果：

书中强调 the ordering of libraries and object files on the command line is significant, 然而没说链接算法是不是有问题，所以我们要注意库和目标文件的排列顺序，细想似乎也没有其他更好的方法

试了试其他 linker 结果也一样

-fuse-ld=bfd
Use the bfd linker instead of the default linker.
-fuse-ld=gold
Use the gold linker instead of the default linker.

C++ 可以通过 namespace 来解决这个问题，库的作者应该把代码写到库的命名空间里面。C 也可以实现命名空间，可以通过命名来实现，比如

void libsub_myfun();

参考  Why doesn't ANSI C have namespaces? - Stack Overflow

也可以用 struct 实现：

// sub/myfun.h
#ifndef myfun_h
#define myfun_h
struct somelib {
    void (*myfun) (void);
};

extern const struct somelib somelib;
#endif

// sub/myfun.c
#include "myfun.h"
#include <stdio.h>

void hello(void) { printf("myfun in libsub.a\n"); }

const struct somelib somelib = {.myfun = hello};

// sub.c
#include "sub.h"
#include "myfun.h"
int sub(int a, int b)
{
    somelib.myfun();
    return a - b;
}

结果：

$ ./a.out
myfun in libadd.a
myfun in libsub.a

参考：For completeness there are several ways to achieve the “benefits” you might get from namespaces, in C.

所以库的作者应该避免这种情况。对用户而言，如果不小心用了没有命名空间保护的库，应该可以快速写一个适配器解决。
