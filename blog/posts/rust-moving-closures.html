Moving A Rust Closure

Recently I was writing a supposedly simple Rust module, there was a lot of repeation so I thought perhaps I should writing code that generate code. I ended up passing a closure into another function that generates another closure. The code won't compile and I don't mind if the design is wrong, I determind to find out why this logically sound operation is not permitted.

fn many_internal<'a, I, L>(
    lookahead_fn: L,
    input: I,
    state: State,
) -> Result<(Option<Vec<OptionItem>>, I, State)>
where
    I: Iterator<Item = &'a String> + 'a + Clone,
    L: Fn(I, State) -> Result<(Option<OptionItem>, I, State)>,
{
    if is_end_of_tokens(&input) {
        return Ok((None, input, state));
    }

    let (opt, i, s) = lookahead_fn(input, state)?;
    match opt {
        None => Ok((None, i, s)),
        Some(opt) => {
            let (opts, i, s) = many_internal(lookahead_fn, i, s)?;
            match opts {
                Some(mut opts) => {
                    opts.push(opt);
                    Ok((Some(opts), i, s))
                }
                None => Ok((Some(vec![opt]), i, s)),
            }
        }
    }
}


fn many<'a, I, L>(
    lookahead_fn: L,
) -> impl Fn(I, State) -> Result<(Option<Vec<OptionItem>>, I, State)>
where
    I: Iterator<Item = &'a String> + 'a + Clone,
    L: Fn(I, State) -> Result<(Option<OptionItem>, I, State)>,
{
    move |input: I, state: State| -> Result<(Option<Vec<OptionItem>>, I, State)> {
        many_internal(lookahead_fn, input, state)
    }
}

The error says

use of moved value: lookahead_fn: let (opts, i, s) = many_internal(lookahead_fn, i, s)?;

That means although many_internal looks like the first function to consume lookahead_fn, in fact it's not, for reasons I can't imagine.

Then I turned to the community for help, some one replied: try passing lookahead_fn by reference.

The code was changed to:

        many_internal(&lookahead_fn, input, state)

Surprisingly it compiles, but I didn't know why, as I was trying to explain it to myself, another person replied: you are wrong from the start.

I forget what the exact words he said, but he pointed out that the closure that I want to generate is defined to be used multiple times.

Then I understood, there is a similar closure type FnOnce which guarantees to only run once, Fn doesn't do that, so the second time it runs, lookahead_fn has already been moved.

The compiler doesn't care if you actually use the closure only once, if you intend to do that you should use the keyword FnOnce which is the exact meaning of "I swear it's an once-off closure". You can't be ambiguous.

To understand why a reference can be moved after the referee is gone, checkout out this document:

Additionally, for any type F that implements Fn, &F implements Fn, too

I never mean to use a reference, you see, my intention is to move the entire lookahead_fn into a closure, so that it can be passed on, to avoid being destroyed by many().

But now error is shifted to many_internal, because it can't use lookahead_fn twice. many_internal is a recursive function, the first call to it causes lookahead_fn to be moved.

&loadhead_fn can't survive the function

Let's start over again, remove the move, and see what happen.


fn many<'a, I, L>(
    lookahead_fn: L,
) -> impl Fn(I, State) -> Result<(Option<Vec<OptionItem>>, I, State)>
where
    I: Iterator<Item = &'a String> + 'a + Clone,
    L: Fn(I, State) -> Result<(Option<OptionItem>, I, State)>,
{
    |input: I, state: State| -> Result<(Option<Vec<OptionItem>>, I, State)> {
        many_internal(&lookahead_fn, input, state)
    }
}

The error says:

the parameter type L may not live long enough, consider adding an explicit lifetime bound L: 'static...

lookahead_fn does not live long enough, borrowed value does not live long enough, that's understandable as many() returns, lookahead_fn is destroyed.

Move into many_internal

fn many<'a, I, L>(
    lookahead_fn: L,
) -> impl Fn(I, State) -> Result<(Option<Vec<OptionItem>>, I, State)>
where
    I: Iterator<Item = &'a String> + 'a + Clone,
    L: Fn(I, State) -> Result<(Option<OptionItem>, I, State)>,
{
    |input: I, state: State| -> Result<(Option<Vec<OptionItem>>, I, State)> {
        many_internal(lookahead_fn, input, state)
    }
}

The error:

cannot move out of lookahead_fn, a captured variable in an Fn closure
move occurs because lookahead_fn has type L, which does not implement the Copy trait

The English phrase "move out of" is confusing, read this one: https://doc.rust-lang.org/error-index.html#E0507

This can also happen when using a type implementing Fn or FnMut, as neither allows moving out of them (they usually represent closures which can be called more than once). Much of the text following applies equally well to non-FnOnce closure bodies.

Rephrase: move out of them, they re closures

Also: moving a member out of a mutably borrowed struct will cause that error

It's confusing, I believe it's a mistake, you can say "I move out of the house", "I" am not in the house any more, but what's moved out of lookahead_fn？ Is it a language misuse？

Look at this example:

let mut sort_operations = vec![];
    let value = String::from("by key called");

    list.sort_by_key(|r| {
        sort_operations.push(value);
        r.width
    });

The closure captures value then moves value out of the closure by transferring ownership of value to the sort_operations vector.

"cannot move out of value", now we are doubly sure that this is a misinformation.

The error message is misleading, however, judging from the rest of the document, we can see that what it actually means is that the value variable can't be moved out from the closure. In the case of many(), the lookahead_fn can't be moved out from many_internal(), therefore it cannot be reused in many_internal(). That's reasonable, you can't move a consumed variable back to the caller. Later I learn that if lookahead_fn has the Copy trait, then when it fails to be moved, the compiler will copy it, but that is another subject.

Now it's time to reconsider the design. The complexity of closures generating closures is flipside of the fancy. Closure can't be copied, that inspires me to use a plain struct which implements the Copy trait but carries the heavy payload by reference. In the next article I will show you how references add to the difficulty of code, and why in the end I have to ditch them.
