Differences between vector and list

I have been through several interviews, and was asked to tell the difference between vector and list 3 or 4 times. Normally I would say vector uses a continous storage and employs a 2x storage growth strategy, and list uses a linked list, which performs better when much insertion and deletion is required at random postions. But it is not satisfying, so I decided to read the GCC source.

Looks like nightmare. First of all, a reserve(n) is called, the function will make sure n is not larger than max_size(), that is to say, the user case of reserver happens when you try to shrink vector's capacity.(as mentioned later, max_size() is like INT_MAX)

If n is legal, then vector performs some memory operations and finally marks the end of storage to be _M_start + n, easy to understand.

I suppose _M stands for memory, no? _M_impl wraps the raw memory, Let's call it _M for short. _M_allocate_and_copy as its name implies, allocates, then copies from _M, from start to end, then destroys objects in _M calling their destructors, and finally deallocates!

At first I was wondering why allocate new memory instead of just trimming the tail; Such Foolish thought. Because, any chunk you allocate must be deallocated as a whole entity. When you reserve, you want to save memory(not necessarily save, but perhaps enlarge), there is no choise but to use a small new one.

00132     typename vector<_Tp, _Alloc>::iterator

00134     erase(iterator __position)
00135     {
00136       if (__position + 1 != end())
00137     _GLIBCXX_MOVE3(__position + 1, end(), __position);
00138       --this->_M_impl._M_finish;
00139       this->_M_impl.destroy(this->_M_impl._M_finish);
00140       return __position;
00141     }

Use _GLIBCXX_MOVE3 to move objects behind the item to be erased backwards, like bubbling, that item will bubbled up to the last position, finally be destroyed.

Let's checkout how it inserts: libstdc++-v3/include/bits/list.tcc

  template<typename _Tp, typename _Alloc>
    typename list<_Tp, _Alloc>::iterator
    list<_Tp, _Alloc>::
    insert(const_iterator __position, const value_type& __x)
    {
      _Node* __tmp = _M_create_node(__x);
      __tmp->_M_hook(__position._M_const_cast()._M_node);
      this->_M_inc_size(1);
      return iterator(__tmp);
    }

the hook operation:

    void
    _List_node_base::
    _M_hook(_List_node_base* const __position) _GLIBCXX_USE_NOEXCEPT
    {
      this->_M_next = __position;
      this->_M_prev = __position->_M_prev;
      __position->_M_prev->_M_next = this;
      __position->_M_prev = this;
    }

insert one single value, steps:

  • create_node
  • connect prev and next using hook
  • done

Conclusion

  • When inserting a sequence or one element into a vector, the vector will allocate new memory and copy old elements, it must be expansive. While list insertion simply takes a node creation
  • Vector's memory is allocated as a chunk, when memory is insufficient, new length is calculated as size() * 2 or size() + new_len if new_len is larger. As for list, it's not hard to imaging that not extra efforts is required, simply _M_create_node
  • Almost forgot one last thing, the memory access, see the code:

vector:

      const_reference
      operator[](size_type __n) const _GLIBCXX_NOEXCEPT
