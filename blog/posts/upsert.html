Database Upsert

Upsert is:

  • a DBMS feature[1]
  • used to atomically either insert a row, or update the row if it already exists[1:1]
  • a clause that causes the INSERT to behave as an UPDATE or a no-op if the INSERT would violate a uniqueness constraint[2]
  • not standard SQL[2:1]

The reason I am concerned in this database feature is that in a project I am working at, I want to implement a insertion functionality that if the entry I insert collides with another one in some aspects, then the insertion should be cancelled and the timestamp column of the existing entry should be updated so as to trace the access, otherwise a new entry is inserted into the table.

Before I knew upsert is an option, I wrote code that checks for collision, if none exists, then the code make an insertion request. This will give rise to race condition, as I later learn from the Ecto document[3]. The document states the problem in great detail and worths reading. Later then I decide to apply the feature the document advocates to my project, and here is the code:


  def create_bookmark(%{"url" => url}) do
    timestamp =
      NaiveDateTime.utc_now()
      |> NaiveDateTime.truncate(:second)

    on_conflict = [set: [updated_at: timestamp]]

    %Clip.Post{}
    |> Clip.Bookmark.changeset(%{title: url})
    |> @repo.insert(on_conflict: on_conflict, conflict_target: :guid)
  end

As the code is very specific to my project, I run a quick test to see how UPSERT works in general terms with sqlite3 /tmp/test.db:

CREATE TABLE IF NOT EXISTS "posts" ("id" TEXT_UUID NOT NULL PRIMARY KEY, "url" TEXT, "inserted_at" TEXT_DATETIME NOT NULL, "updated_at" TEXT_DATETIME NOT NULL);

CREATE UNIQUE INDEX "posts_url_index" ON "posts" ("url");

Notice that I set a uniqueness constraint on url column

INSERT INTO "posts" AS b0 ("url","id","inserted_at","updated_at") VALUES ("https://www.test.com/", "0e47", "t1", "t1") ON CONFLICT ("url") DO UPDATE SET "updated_at" = "t2";

INSERT INTO "posts" AS b0 ("url","id","inserted_at","updated_at") VALUES ("https://www.test__2.com/", "0e47", "t1", "t1") ON CONFLICT ("url") DO UPDATE SET "updated_at" = "t2";

This throws an error:

Error: stepping, UNIQUE constraint failed: posts.id (19)

The error says that the uniqueness of the primary key id is violated, but of course it is violated, why should I insert a entry with the same primary key? In practise the id is auto-generated, and is unlikely to be identical to the ones generated before.

Take a look at the table:

sqlite> select * from posts;
0e47|https://www.test.com/|t1|t1

Let's try something practical this time, create a new insertion with a unique id :

INSERT INTO "posts" AS b0 ("url","id","inserted_at","updated_at") VALUES ("https://www.test.com/", "0e48", "t1", "t1") ON CONFLICT ("url") DO UPDATE SET "updated_at" = "t2";

The url of this new entry conflict with another entry in the table, however, the database does not alert or abort the operation, instead it proceeds and update the updated_at column to t2

sqlite> select * from posts;
0e47|https://www.test.com/|t1|t2

Check the database, the entry with https://www.test.com/ still stick to id 0e47, nothing changes except t2 which is updated on conflict.

So, yes, this is what I want. SQLite follows the upsert syntax used by PostgreSQL[4], I feel lucky not to have chosen MySQL as it does not support "conflict target"[5].

But there is also a problem: I have no idea which operation takes place during the "UPSERT", insert or update?

      Process.sleep(1000)

      {:ok, post2} =

]

This problem is also stated in the Ecto document[6]. I did not expect the return data was not the one that reflected the actual data in database, but the one that the code made up for insertion. As a result, users of the insert API must be very careful.

One other thing: the difference between "insert_or_update" and "upsert", to use the former, you must load existing structs from the database if you want to update them, this involves at least 2 requests, and more logical decisions, whereas "upsert" is more direct and firm, you only need to issue one request to execute it.
