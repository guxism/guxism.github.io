虚拟网卡实验

创建 tun

tun 是用来转发 IP 层 payload 的

首先像打开任何文件那样打开一个描述符

char *clonedev = "/dev/net/tun";
fd = open(clonedev , O_RDWR);

然后用 ioctl 设置它的属性：这里，fd 是 device file，kernel 的 device driver 需要将进程的输出发送给设备，或者将设备的输入发送给进程，进程包含 fd，在进程中设置 fd, 即是与内核交互，让内核认识一个 fd 的存在

Every device has its own ioctl implementation.

比 ioctl 更好的方法：

  • system calls
  • netlink
  • debugfs
  • sysfs
  • configfs

为什么说 netlink 是更好的方法呢，不清楚，但可以参考为什么 ip link 能取代 ifconfig, 前者用的就是 netlink

参考这： https://patchwork.ozlabs.org/project/netdev/patch/1241433778.6126.90.camel@macbook.infradead.org/

在 R2S 创建 tun

ip tuntap add dev tun2 mod tun
ip link set tun2 up
ip addr add 10.11.12.2/24 dev tun2

然后 ping 10.11.12.2, 监听 tun2, 看不到任何包

如果 ping 10.11.13.2 (eth0) 并监听：tshark -i eth0, 一样看不到任何结果，

root@nanopi-r2s:~# ip addr add 10.11.13.2 dev eth0
root@nanopi-r2s:~# tshark -i eth0

为什么呢？因为内核看到这个地址不需要发送到外部(on the wire)，那么内核就自动回应了 ping，这是可以理解的

我在监听 lan0 也就是和外界通讯的端口，然后 ping 192.168.1.2

内核判断 192.168.1.2 不在本机，所以就要在 wire 上查找(on the wire)

但我 ping 10.11.12.1 时候，内核并不发出 ARP 请求

The kernel sends packets out of the tun interface if the destination address is not local and the existing routing table tells it to do so

root@nanopi-r2s:~# ip route show all
default via 192.168.1.1 dev lan0 proto dhcp metric 100 
10.11.12.0/24 dev tun2 proto kernel scope link src 10.11.12.2 linkdown 
169.254.0.0/16 dev lan0 scope link metric 1000 
192.168.1.0/24 dev lan0 proto kernel scope link src 192.168.1.3 metric 100

It's simple: with older kernels, the interface was always considered to be up, so one could just throw anything out of it and tcpdump would get it. Starting from kernel 2.6.36, a change was added that sets the interface "up" only when there is a program connected to the interface, which is effectively the equivalent of a "carrier" or a "wire" for the tun interface. It's no different from what you see with a regular ethernet interface: if the cable is disconnected, you get a "no-carrier" status. With this new feature, it's no longer possible to ping an address belonging to the tun interface subnet and see the packets going out, unless a program has previously been attached to the interface.

https://backreference.org/2010/03/26/tuntap-interface-tutorial
