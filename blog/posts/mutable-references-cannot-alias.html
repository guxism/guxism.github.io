From Mutable Reference Aliases To Interior Mutability

  1. The rationale for disallowing mutable aliasing
  2. Enforce the borrowing rules at runtime instead of compile time

The rationale for disallowing mutable aliasing

By default, the Rust compiler disallow mutable reference aliases[1][2][3]

The compiler only optimize &mut references if and if only they are exclusive. This answer[3:1] provides good explaination, in short:

if mutable references are allowed to alias, then the compiler has no idea, or need more details to tell whether or not some different references are pointing to one object. The compiler would easily optimize code with exclusive references. But not the other way around.

Here is an example:

fn foo(r1: &mut i32, r2: &mut i32) {
	*r1 = 1;
	println!("{}", *r1);
	*r2 = 2;
	println!("{}", *r2);
}

The compiler could rearrange the IR(Intermediate Representation) making the code above to have the same effect as the code below:

fn foo(r1: &mut i32, r2: &mut i32) {
	println!("{}", 1);
	println!("{}", 2);
	*r1 = 1;
	*r2 = 2;
}

The compiler thinks that dereferencing has a loading cost, but that is totally fine to get rid of, so it uses equivelent intermediates, in this case, 1 and 2. So far, so good, but suppose r1 and r2 are referencing to the same object, then the optimized code above is not that optimized, it can be optimized further like this: (since r1 and r2 refer to the same object, you change the object's value through one of them, the other reference will still point to that object without changing something)

fn foo(r1: &mut i32, r2: &mut i32) {
	println!("{}", 1);
	println!("{}", 2);
	*r2 = 2;
}

But how does the compiler know if r1 is aliasing r2? As I suggested before, the compiler doesn't know. If more information is needed to perform the optimization, then I believe the Rust compile will take that informatio, however slower the compilation would become, because it has been slow enough so that it doesn't mind being slower. But since the compile forbids mutable aliasing, we can infer that the compile does not own that kind of information.

Enforce the borrowing rules at runtime instead of compile time

This official statement is not easy to find.

Ref<T> and RefMut<T>, accessed through RefCell<T>, a type that enforces the borrowing rules at runtime instead of compile time -- https://doc.rust-lang.org/book/ch15-00-smart-pointers.html

Although Rust is strict but I guess there are always exceptions for rules that are invented by human. RefCell makes an exception for borrowing rules enforcement that is supposed to happen in compile time and postpone it to runtime.

Let's take a look at how RefCell implements the so called "interior mutability" design pattern.

trait Immutable {
    fn never_mutate(&self);
}

struct MutableStruct {
    data: i32
}

impl Immutable for MutableStruct {
    fn never_mutate(&self) {
        self.data += 1;
    }
}

fn main() {

}

The code is what it is, self explained. So this code won't compile, unless you change it to:

use std::cell::RefCell;

trait Immutable {
    fn never_mutate(&self);
}

struct MutableStruct {
    data: RefCell<i32>
}

impl Immutable for MutableStruct {
    fn never_mutate(&self) {
        *self.data.borrow_mut() += 1;
    }
}

fn main() {

}

We see that RefCell wraps the references and provide borrow() and borrow_mut() interfaces for the outside to to interact with its interior.

Now we want multiple mutable references:

use std::cell::RefCell;

trait Immutable {
    fn never_mutate(&self);
}

struct MutableStruct {
    data: RefCell<i32>
}

impl Immutable for MutableStruct {
    fn never_mutate(&self) {
        *self.data.borrow_mut() += 1;
    }
}

fn main() {
    let a = MutableStruct {
        data: RefCell::new(1)
    };
    let mut one_borrow = a.data.borrow_mut();
    let mut two_borrow = a.data.borrow_mut();
    *one_borrow = 2;
    *two_borrow = 3;

}

The code compiles but throws an error in runtime:

thread 'main' panicked at 'already borrowed: BorrowMutError', examples/15-5-refcell.rs:22:33
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

The workaround is to let a reference counter guard the interior:

use std::{cell::RefCell, rc::Rc};

trait Immutable {
    fn never_mutate(&self);
}

#[derive(Debug)]
struct MutableStruct {
    data: Rc<RefCell<i32>>,
}

impl Immutable for MutableStruct {
    fn never_mutate(&self) {
        *self.data.borrow_mut() += 1;
    }
}

fn main() {
    let a = MutableStruct {
        data: Rc::new(RefCell::new(1)),
    };
    let one_rc = a.data.clone();
    let two_rc = a.data.clone();
    let three_rc = a.data;
    *three_rc.borrow_mut() = 2;
    dbg!(&one_rc);
    dbg!(&two_rc);
    *one_rc.borrow_mut() = 3;
    dbg!(&one_rc);
    dbg!(&two_rc);
}

We can't actually borrow_mut() from one reference twice, Rust let you escape the compile time enforcement, but even in runtime mutable sharing is not going to happen. Functional programming paradigm says that mutable states are evil. Rust tries its best to keep us from them. Now each *_rc is holding a mutable reference, each one is able to mutate the same object, that implements the so called mutable aliasing but in a very twisted way. It seems natural to have multiple mutable references. But don't forget that in the beginning how Rust snuffs out our attempt to mutably alias an object twice, when we in fact don't intent to actually modify something, simply writing such code is illegal enough:

fn main() {
    let a = 2;
    let r1 = &mut a;
    let r2 = &mut a;    
}

Now by adding more rules Rust allows us to break the previous rule, and we have arrived at the place where we can alias mutable references, in the simplest form, that is:

fn main() {
	let a = 2;
	let r1 = SOME_WORKAROUND(a);
	let r2 = SOME_WORKAROUND(a);
}

Conclusion

I believe we had spent too much time in this topic, we need to put an end to it because the more we think about it, the more confused we will get. Just remember the reason mutable aliasing is not allowed simply because Rust can't perform optimization on it, because there is not enough information to tell if different mutable references are referring to the same thing. But time has changed, Rust provide more syntax to annotate mutable aliasing, that is rc<RefCell<T>>, on account of which the compiler will keep its optimization for references not pointing to the same object, and whether the compile has plan for mutable aliasing is not my concern.

  1. In Rust, mutable references never alias ↩︎

  2. &mut references cannot be aliased, even when using unsafe, in order for your code to have defined behavior. ↩︎

  3. The compiler is allowed to optimize &mut pointers under the assumption that they are exclusive (not aliased) ↩︎ ↩︎
