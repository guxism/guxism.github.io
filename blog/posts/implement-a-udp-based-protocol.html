Implementing A UDP-Based Reliable Protocol

Once upon a time my job involved me into a project that implemented a reliable UDP-based RPC-like protocol. I am going to talk about it because we keep talking how unreliable UDP is, but someone useed it to archive reliability nonetheless. I didn't designed the protocol, but I am obliged to provide a data formation.

The first step was to let both ends know when to begin, so there should handshakes. TCP uses a three-way handshake procedure, as stated in RFC 793, the purpose is to keep the server from old duplicate SYNs. But in this project, the protocol proposed a two-way handshake. I Guess it was because that was easy to implement, and the circuit where network traffic went through was short enough to rule out packet delays. In short, the protocol is context specific and is used for one purpose only. Otherwise, it's impossible for us to argue against the TCP protocol which has undergone realistic tests along the years.

The point is, what TCP considers to be the best effort to be realiable is, to some specific purposed protocol, overreacting.

Then I built a state machine. Reliable connections are "stateful". Each time the server receives a message, it first looks at the its current state and then decides what to do with the message. If the message is legal, the server responds with an ACK; otherwise, it returns an error. The client will be blocked until it receives the feedback of the last packet or a timeout occurs. The client will retry several times and give up if retries all fail.

Meanwhile the recipient should maintain a sequence number to keep data in order and avoid duplicates.

The final step is to terminate the connection. When the recipient receives a final signal, it reassembles the data, processes it, and then terminates the connection.

Now we have come to the point to wonder why not use TCP in the first place. The truth was surprising, the system used multicast to connect with any possible recipients but it has no prerequisite knowledge of what recipients they are, such is the design of the system, and multicast is not possible with TCP. As I recall the project, I realize that the design where two devices communicate via multicasting is somehow weird. They should design a registration center, but that was not up to me.

As a matter of code, I wrote some like this:

struct packet
{
    uint8_t is_last_pkt;
    uint8_t seq;
    uint16_t len;
    uint8_t *msg;
};

struct buffer
{
    uint8_t done;
    uint8_t seq;
    struct packet pkts[MAX_SEQ];
} buf;

Some observation:

  1. The switch we used in the project truncated packets that passed through it, causing them to lose bytes. TCPdump on the server can receive these truncated packets; UDP sockets can not
  2. Checksums of the UDP packets received by the server are always empty
  3. We observed duplicated multicast packets. The switch caused the problem.

It is interesting to observe and take part in publishing and implementing a protocol that is real, though a lot less general and robust, unlike those we study and memorize for job iterviews. To me the experience was priceless.
